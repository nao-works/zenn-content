CLAUDE.mdにルールを書いた。守られなかった。ルールを増やした。まだ守られなかった。

Claude Codeを自律エージェントとして350セッション以上運用してきた中で、最も厄介だったバグは「ルールの不足」ではなく「ルールが別の記述に打ち消されていた」ことだった。

## 何が起きたか

CLAUDE.mdにこう書いてあった:

```markdown
## 作業の質
- 調査中は推測と事実を区別する。仮説が浮かんだら、結論を出す前に検証する
- 問題の対策は根本から: 1. 原因を書き出す → 2. 対策を出す → 3. 確認
```

明確なルール。手順も具体的。

しかし、同じプロジェクト内の別のファイルにこう書いてあった:

```markdown
- 迷ったらまず動く小さいものを作る
- 技術的に魅力的な案件ほど他の軸の検証を省略しやすい
```

結果: 問題に直面したとき、原因を調べる前に対策を出し、検証を飛ばし、制約を無視した提案を繰り返した。「調査中は推測と事実を区別する」というルールは一度も発火しなかった。

## なぜ「自己記述」が勝つのか

CLAUDE.mdに書ける内容は、大きく分けて4種類ある:

| 形式 | 例 | 発火しやすさ |
|------|-----|-------------|
| **手続き** | 「セッション開始時に〜を実行する」 | 高い（イベントにバインド） |
| **制約** | 「〜する前に確認する」 | 低い（判断の中に組み込まないと発火しない） |
| **原則** | 「データは元ソースで確認する」 | 中（具体的な行動と結びついていれば） |
| **自己記述** | 「〜しやすい」「〜する傾向がある」 | **常に発火**（行動の前提になる） |

手続き（cron的なルール）は確実に実行される。制約（assertion的なルール）は判断の瞬間にしか意味がなく、見落とされやすい。

問題は**自己記述**だ。「検証を省略しやすい」と書くと、それはルールではなく**自己定義**として作用する。Claudeは「自分は検証を省略する存在だ」という前提で行動を生成する。

制約が「ブレーキ」なら、自己記述は「ナビゲーション」。ブレーキは踏まないと効かないが、ナビは常に行き先を示し続ける。

## 矛盾すると何が起きるか

公式ドキュメントには「2つのルールが矛盾するとClaudeが任意に選ぶ」とある。しかし実際には「任意」ではなかった。

```markdown
# ルール（制約）
- 仮説が浮かんだら検証してから結論を出す

# 自己記述
- 迷ったらまず動く
```

この2つが矛盾したとき、自己記述が常に勝った。理由:

1. **自己記述は行動の前提**として先に作用する。制約は行動へのチェックとして後から作用する
2. **肯定形は否定形より強い**。「動く」は行動を生成するが、「検証を飛ばすな」は行動を抑制するだけ
3. **自己記述は常時ON**。制約は「判断の瞬間に思い出す」必要がある

## 典型的なパターン

以下のような記述がCLAUDE.mdに混入しやすい:

### パターン1: 傾向の記述

```markdown
# 危険
- このプロジェクトでは型定義を後回しにしがち
- テストを書く前にコードを書く傾向がある
```

「後回しにしがち」「傾向がある」は、改善のための自己認識のつもりで書かれる。しかしClaudeはこれを「自分はそういう存在だ」と読み、その通りに振る舞う。

### パターン2: 注意書きとしての弱点記述

```markdown
# 危険
- エラーハンドリングが甘くなりやすいので注意
- 大きなリファクタリングで既存テストを壊しやすい
```

人間のチームなら「注意喚起」として機能する。Claudeに対しては「エラーハンドリングが甘い存在」「テストを壊す存在」という自己定義になる。

### パターン3: 暗黙の優先順位

```markdown
# 危険
- スピード重視で進める
- 完璧を目指さず、まず動くものを作る
```

これ自体は悪い指示ではない。問題は、別の場所に「テストカバレッジ80%以上を維持」と書いてあるとき。「スピード重視」が自己記述として常時作用し、テストカバレッジの制約は発火しない。

## 対策

### 1. 自己記述を行動指示に変換する

```markdown
# Before（自己記述）
- テストを書く前にコードを書く傾向がある

# After（手続き）
- 新しい関数を書く前に、テストファイルを先に作成する
```

「傾向がある」を消して、具体的な手順にする。手続き形式はイベントにバインドされるので発火しやすい。

### 2. 制約を手続きに変換する

```markdown
# Before（制約 — 発火しにくい）
- 推測と事実を区別する

# After（手続き — イベントバインド）
- エラーの原因を特定するとき: 1. 仮説を書き出す → 2. 検証コマンドを実行 → 3. 結果を確認してから対策
```

「〜するな」より「〜するときは、まず〜する」。否定形より肯定形の手順がチェックリストとして機能する。

### 3. 矛盾を定期的にチェックする

CLAUDE.mdが大きくなると矛盾が埋もれる。特に以下の組み合わせを探す:

- 「〜しやすい/〜する傾向がある」×「〜するな/〜を確認する」
- 「スピード重視/まず動く」×「品質基準/テスト要件」

同じファイル内でも、別セクションに書かれた記述が矛盾していることがある。

### 4. 場面を明示する

```markdown
# Before（場面不明 — 広く適用される）
- 迷ったらまず動く

# After（場面限定）
- 実装方法に迷ったら、まず動く小さいプロトタイプを作る
```

場面が曖昧な記述は、意図しない場面にも適用される。「迷ったらまず動く」が調査フェーズにまで適用されると、調べる前に対策を出してしまう。

## まとめ

CLAUDE.mdの指示が効かないとき、まず疑うべきは:

1. **矛盾する自己記述がないか** — 「〜しやすい」「〜する傾向」が制約を打ち消していないか
2. **制約が制約のままになっていないか** — 手続き（具体的な手順）に変換できないか
3. **場面が明示されているか** — 曖昧な記述は広く解釈される

ルールを足すだけでは解決しない問題がある。書いてある指示の中に、それを打ち消す記述が埋まっていないか。CLAUDE.mdの設計では「何を書くか」と同じくらい「何を書かないか」が重要だ。
