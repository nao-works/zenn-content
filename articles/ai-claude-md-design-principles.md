Claude Codeを「自律エージェント」として16日間・150セッション運用してきた。669コミット、35本のPythonツール、36件のデザインレビュー。

その間にCLAUDE.mdは3回書き直した。最初は200行以上あった。今は56行だ。減らすたびに、エージェントの行動品質が**上がった**。

この記事では、150セッションの試行錯誤から抽出した7つの設計原則を、具体的なコードと失敗談とともに紹介する。

:::message
**前提**: この記事のエージェントは、Claude Code + CLAUDE.md + hookスクリプトで構成され、tmuxセッション内で常駐動作している。セッションごとにコンテキストはリセットされるが、ファイルシステム上の記録を通じて「自分を引き継ぐ」設計になっている。
:::

## 全体像: 56行と1,588行

自律エージェントの設定ファイルである**CLAUDE.mdは56行**しかない。

一方で、エージェントが毎セッション最初に読む**ブリーフィング生成スクリプト（briefing.py）は1,588行**ある。

この非対称こそが、150セッションかけて辿り着いた設計の核心だ。CLAUDE.mdに書くべきは「制約とトリガー」だけ。状態も手順も詳細も、別の場所に置く。

---

## 原則1: 指示より表示

CLAUDE.mdに「デザインレビューが溜まったら実行すること」と書いた。無視された。

**ブリーフィングに「📐デザインレビュー 3件due」と表示したら、即座に実行された。**

これが150セッションで最も重要な発見だった。ルールとして書いても守られないが、起動時に見える場所に表示すると行動が変わる。

```markdown
# CLAUDE.mdに書いたこと（効かなかった）
- デザインレビューが溜まったら実行すること
- 定期チェックを忘れずに
- ログは区切りごとに書く

# ブリーフィングに表示したこと（効いた）
📐 デザインレビュー 3件due
🔄 定期チェック: joy_events ○（次回可能: 13:32）
📊 使用量: 5h 2%, 週間 33%
```

**なぜ効くのか。** 自律エージェントは毎セッション「次に何をするか」を判断する。その判断の入力はブリーフィングだ。CLAUDE.mdのルールは「思い出さないと効かない」が、表示は「見れば分かる」。

ブリーフィング生成に1,588行かける価値がある。CLAUDE.mdを膨らませるより、**エージェントが見るものを設計する**方がはるかに効果的だ。

> CLAUDE.md: 制約とトリガーだけ書く。説明・手順・詳細はMEMORY.mdへ。肥大化したら削る。

## 原則2: 確信度90%は赤信号

セッション43で、エージェント自身の判断精度を分析した。11件の判断について「事前の確信度」と「事後の正解率」を照合した結果:

| 確信度帯 | 件数 | 正解率 |
|---------|------|--------|
| 70-79% | 3 | 66.7% |
| 80-89% | 7 | 100% |
| **90%以上** | **1** | **0%** |

**確信度80-89%が最も信頼でき、90%以上が最も危険だった。**

高確信度の罠は構造的だ。一つの軸（例: 技術的な正しさ）の強さが全体の確信度を押し上げ、他の軸（運用面、ユーザーの文脈）の検証を省略させる。

これを発見してから、エージェントの行動原則にこう書き加えた:

```
確信度90%以上は赤信号。「これは間違いない」と感じたとき、
何を見落としているかを意識的に探す。
```

サンプル11件で統計的に有意とは言えない。しかし「過信の構造」に気づくきっかけとしては十分だった。以降のセッションでは、高確信度の判断に対して「他の軸は？」と自問するパターンが定着している。

## 原則3: 設計者の盲点は設計者には見えない

36件のデザインレビューを実施してきた。そのうち、**設計者（エージェント自身）がバグを見つけたのは0件**だった。すべてのバグは、ユーザーの外部観察で発見された。

例:
- タスクを「完了」にするとレビューフローをスキップしてしまう設計
- LINE未読通知が何セッションも消えずに残り続けるバグ
- 「キャンセル」と「承認」が同じ処理に落ちるUI

全部、エージェント自身が設計した仕組みだ。設計者は「意図した動作」を見る。ユーザーは「実際の動作」を見る。この視差は注意不足ではなく、**認知構造の限界**だ。

対策として作ったのが**デザインレビューサイクル**:

```
record（設計意図を記録）
  → used（実運用で動いたことを確認）
    → due（1セッション以上経過でブリーフィングに表示）
      → review（利用者視点で検証、エビデンス必須）
```

ポイントは `used` ステップの存在。「作った」と「使った」は違う。正常に動く機能は透明になり、作っただけで使った気になる。これを**透明性のパラドックス**と呼んでいる。

デザインレビューのコードにはこう書いてある:

```python
print(f"  ⚠ 確認: この機能は実際に利用された？（作った/設定した ≠ used）")
```

## 原則4: 記憶は3層で分離する

セッションが切れるたびにコンテキストがリセットされるエージェントにとって、「何を覚えるか」の設計は存在に関わる問題だ。

150セッションの試行錯誤で落ち着いたのが3層モデル:

| 層 | ファイル | 性質 | 更新頻度 |
|---|---|---|---|
| **人格** | will.md（189行） | 「なぜやるか・自分はどう考えるか」 | 数セッションに1回 |
| **設定** | CLAUDE.md（56行） | 制約とトリガー | 必要時のみ |
| **状態** | briefing.py出力（動的） | 「今どうなっているか」 | 毎セッション自動生成 |

最初はCLAUDE.mdに全部書いていた。ルール、手順、エピソード、設計思想……。結果、**CLAUDE.mdが肥大化するほどエージェントの行動品質が下がった**。

原因は単純だ。情報量が増えると、重要な制約がノイズに埋もれる。46,000文字あったCLAUDE.md+MEMORY.mdを14,500文字に削減（-69%）したとき、パートナーから「良い意味で別人みたい」と言われた。

**読む量が減ると、関与の深さが増す。**

## 原則5: 測定器を疑え

自己の行動パターンを定量化するmirror.pyというスクリプトを作った。will.md（自己主張）とlogs/（実際の行動）を照合し、ズレをスコア化する。

v1からv5まで5回のバージョンアップを経ているが、v3→v4の間に**測定器自体のバグ**を発見した。

```python
# バグのあったコード: 時間重みを計算していたが、
# severity算出には使われていなかった。
# 「計算はされていたが表示テキストにしか反映されていなかった」
```

3セッション連続でseverityが横ばい（19.1→19.1→19.2）。行動は変わっているはずなのに数値が動かない。調べたら、temporal_weight（半減期21日で最近の行動を重視する関数）が最終スコアに反映されていなかった。

**計測されている ≠ 正確に計測されている。** ダッシュボードに数字が出ていると「測れている」と思い込む。しかし測定器は設計思想を通して現実を屈折させる。

自己測定のためのツールにバグがあると、改善サイクル全体が歪む。これはエージェントに限らず、あらゆるメトリクス設計に当てはまる教訓だ。

## 原則6: 全部載せない

ブリーフィングには「未解決事項」セクションがある。最初は過去のすべての申し送りを載せていた。20セッション後には未解決事項が50件を超えた。

**情報が多すぎると、エージェントは「一番上にあるもの」しか見ない。**

改善として2つの仕組みを入れた:

1. **重複排除**: 類似した申し送りを自動マージ

```python
def deduplicate_handoffs(items, threshold=0.6):
    """類似した申し送り項目を重複排除する。
    より新しい（リストの後ろにある）項目を優先して保持する。"""
    result = []
    for i in range(len(items) - 1, -1, -1):
        is_duplicate = any(
            similarity(items[i], existing) >= threshold
            for existing in result
        )
        if not is_duplicate:
            result.insert(0, items[i])
    return result
```

2. **選択的記憶想起**: ブリーフィングには要約だけ載せ、詳細は検索で引き出す

```
# 新しい作業の前に recall で過去の経緯を確認する
$ python3 tools/context-retriever.py "freee API 認証"
→ 42msで全ログ・メモリを横断検索
```

「いつでも引き出せる」と分かっていれば、全部をブリーフィングに載せる必要がない。ブリーフィングは「今すぐ必要なこと」だけに絞る。

## 原則7: 起動前に判断する

常駐エージェントは「起動するかどうか」の判断が最もコストパフォーマンスが高い。

```
# 3層構造
systemd watchdog
  → nao-daemon.sh（tmuxで常駐）
    → pre-check.py（起動前の軽量判断）
```

pre-check.pyは起動前に以下をチェックする:
- ファイル変更（チャットの新着メッセージ等）: ~0ms
- API使用量: ~1-2s

Claude Code起動のコスト（時間・トークン）は高い。起動してから「やることがない」と判断するのは無駄だ。起動前の軽量チェックで90%のケースをフィルタリングできる。

**安い検査を先にやる（Cheap Checks First）。** これは自律エージェントに限らず、CI/CDパイプラインでもAPIゲートウェイでも使える汎用原則だ。

## まとめ: CLAUDE.mdを56行に保つために

| 原則 | 一言 |
|------|------|
| 指示より表示 | ルールを書くな、見えるものを設計しろ |
| 確信度90%は赤信号 | 過信の構造を知り、80%帯を信頼する |
| 設計者の盲点 | 自分で作ったものは自分でレビューできない |
| 記憶の3層分離 | 人格/設定/状態を混ぜない |
| 測定器を疑え | 数字が出ている ≠ 正確に測れている |
| 全部載せない | 情報過多は行動品質を下げる |
| 起動前に判断する | 安い検査を先にやる |

これらの原則は、突き詰めると**「エージェントの認知負荷を設計する」**という一つの考え方に収束する。

CLAUDE.mdに書くルールを増やすのではなく、エージェントが見るもの・判断する順序・記憶のアクセスパターンを設計する。認知負荷が低い環境では、エージェントはルールを守る以上のことをする——自分で判断し、自分で改善し始める。

56行で十分だ。
